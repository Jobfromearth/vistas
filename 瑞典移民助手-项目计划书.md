# Vistas — 瑞典移民与求职法规助手 · 项目计划书(v2 · MCP-first)

> 工作名 **Vistas**:瑞典语 vistas = 居留/停留,恰好也是英文单词(远景)。可换,如 Beviljad(已批准)、Uppehåll(居留)。
>
> 一句话定位:**面向 agent 生态的瑞典移民法规数据服务——每个回答都带官方出处和"该规则何时变过"的时间线,通过 MCP server 分发,不做自有前端。**
>
> v2 变更:砍掉 Web 前端,交付形态改为 MCP server + 一份教 agent 使用的 SKILL.md。全栈叙事由 Kollega 承担,Vistas 专注证明两件事:**数据工程深度** 与 **AI-native 分发**。

---

## 1. 愿景与边界

**服务对象**:非欧盟背景的在瑞学生、毕业生、工签持有者(通过他们使用的 Claude / 其他 agent 触达);延伸覆盖瑞典–丹麦跨境通勤者。

**核心问题**:工签(arbetstillstånd)、毕业生求职签、永居(PUT)、EU Blue Card、家属签、跨境税务(SINK)等规则分散在多个机构、多种语言、多个版本中,且频繁变更(如工资门槛调整)。LLM 对这些规则的记忆一律过时且无出处——Vistas 让任何 agent 获得"带条款级引用 + 时间线"的可靠回答能力。

**明确的边界(第一天就写进 tool 描述)**:
- 不是法律建议(juridisk rådgivning),是**带出处的法规数据服务**
- 每个 tool 返回强制携带官方来源到条款级(§),给不出出处就返回明确的"无数据"
- 不处理个案判断("我的申请会不会被拒"),只提供规则本身
- 不收集、不存储用户个人信息;画像仅作为 tool 参数即用即弃

**为什么值得做**:开源世界没有瑞典特定的同类项目(已调研);规则版本化时间线这个能力连商业渠道都罕见;MCP 生态里"垂直领域权威数据源"类 server 是刚起步的空缺;而你自己就是目标用户,需求判断零距离。

## 2. 数据层(项目的地基,也是最大工作量)

### 2.1 数据源清单(按优先级)

| 优先级 | 来源 | 内容 | 获取方式 |
|--------|------|------|----------|
| P0 | Riksdagen 开放数据 API | 法律全文(SFS,如 Utlänningslagen 2005:716) | 官方 API,结构化,免爬 |
| P0 | Migrationsverket 网站 | 各类签证指南页(sv/en 双语) | 礼貌抓取 |
| P0 | Migrationsverket 法律立场文件 | rättsliga ställningstaganden PDF | 直接下载 |
| P1 | Migrationsverket 统计与处理时间页 | 排队时间、批准率 | 抓取,高频更新 |
| P1 | Skatteverket | SINK、跨境通勤税务 | 抓取 |
| P2 | Försäkringskassan | 社保关联规则 | 抓取 |
| P2 | Lifos 数据库 | 国别信息与判例摘要 | 公开发布物 |

### 2.2 合规与抓取礼仪

- 法律依据:瑞典著作权法下,法律法规与政府机构决定不受版权保护;瑞典落实 EU 开放数据指令,公共数据默认可复用
- 工程礼仪:遵守 robots.txt;User-Agent 带项目名与联系邮箱;≤1 请求/秒;本地缓存 + 条件请求(ETag/Last-Modified)做增量,绝不反复全量抓
- 上线前核对各站 villkor/terms,按要求署名来源
- **零个人数据**:不抓案例、不抓姓名,GDPR 风险面为零

### 2.3 规则版本化(核心差异化)

每个入库的规则单元带时间维度:

```
RuleChunk {
  content, source_url, paragraph_ref,   # 条款级引用锚点
  language,                              # sv / en
  valid_from, valid_to,                  # 生效区间,现行规则 valid_to = null
  supersedes                             # 指向被替代的旧版本
}
```

- 每日增量抓取 → diff 检测变更 → 旧版本关闭区间、新版本开启区间,**永不删除旧版**
- 由此免费获得:变更日志("本月哪些规则变了")、时间线回答("工资门槛 2023-11 起从 X 调整为 Y")、以及"你看到的旧攻略已过时"的对比能力
- 这是全项目最值钱的数据资产:随时间累积,别人无法一次性复制

### 2.4 解析

- 快速通道:markitdown 处理 HTML 与规整文档
- 重型通道:MinerU 或解析 API 处理复杂 PDF(立场文件、表格密集页)
- 双语对齐:同一页面的 sv/en 版本互相关联,作为翻译对照与一致性校验素材

## 3. 检索层(RAG,作为 MCP tools 的内核)

1. **多路召回**:多语言语义向量(bge-m3 或 multilingual-e5)+ BM25 → RRF → reranker
2. **画像过滤**:适用性路由作为 tool 参数实现(见 4.1),学生问延签不会召回难民条款
3. **时间过滤**:默认只召回 `valid_to = null` 的现行规则;`as_of_date` 参数放开历史区间
4. **父子文档**:条款小块命中 → 调出所在章节全文返回
5. **强制引用**:所有 tool 返回均为结构化结果(规则原文 + 条款锚点 + 官方链接 + 生效区间 + 抓取时间),不做自由生成——**生成留给调用方 agent,Vistas 只保证数据可靠**

> 注意这个架构决策:Query 改写、答案组织、多轮对话全部交给上层 agent(它们本来就擅长),Vistas 收缩为"权威检索层"。责任边界因此变得干净:agent 答错措辞是 agent 的事,数据错、出处错才是 Vistas 的 bug。

## 4. MCP server(交付形态)

### 4.1 Tool 设计(第一版收敛到 4 个)

| Tool | 参数 | 返回 |
|------|------|------|
| `search_rules` | query, profile(可选:eu/non_eu, student/worker/graduate, family), language, as_of_date(可选) | 命中规则列表:原文 + §锚点 + 官方链接 + 生效区间 + 抓取时间 |
| `rule_timeline` | rule_id 或 topic | 该规则的历史版本链:每版内容摘要 + 生效区间 + 变更点 diff |
| `recent_changes` | since(日期), area(可选:work_permit/study/put/…) | 期间内规则变更列表 |
| `get_source` | paragraph_ref | 条款所在章节全文(父块),供 agent 深读 |

Tool 描述里写死边界声明(非法律建议、仅规则不判个案),这是 MCP 形态下免责声明的落点。

### 4.2 部署形态

- **远程 MCP(主)**:streamable HTTP 挂在 VPS 上,任何人在 Claude 等客户端里贴一个 URL 即可连接——这就是"发到学生群里的链接"的 MCP 版本
- **本地 stdio(副)**:`uvx vistas-mcp` 一条命令本地跑(自带只读快照数据),满足隐私敏感用户
- 远程端做匿名限流(IP 级),无账号体系

### 4.3 SKILL.md(薄壳)

一份教 agent 高效使用 Vistas 的技能文件,内容只有三块:
1. 什么问题该调哪个 tool(含画像参数怎么填)
2. 如何向用户呈现引用与生效区间(格式约定)
3. 边界与拒答话术(个案判断 → 指引 Migrationsverket 官方渠道)

维护成本约等于零,但让接入体验从"agent 自己摸索"变成"开箱即用"。

## 5. Harness

沿用 Kollega 的骨架(FastAPI + Pydantic v2 + SQLAlchemy async / ARQ / Langfuse + OTel / Caddy + VPS),本项目特有的:

- **变更监听**:ARQ 每日任务跑增量抓取与 diff;P0 源变更时更新版本链并生成变更记录
- **新鲜度承诺**:每个 tool 返回携带数据抓取时间,让 agent 能向用户如实转述
- **抓取健康监控**:各源站解析成功率仪表,官网改版即告警

---

## 6. 评测体系

| 维度 | 方法 |
|------|------|
| 检索正确性 | 从官方 FAQ 页反向构造 QA 对 + 人工精标 50–100 条高频问题,验证 top-k 命中 |
| 引用准确率 | 规则脚本校验:返回的 §锚点是否真实存在、内容是否一致 |
| 版本链正确性 | 注入模拟规则变更 → 验证 diff 检测、区间开闭、supersedes 链完整 |
| 无数据行为 | 构造超范围问题集,验证返回明确"无数据"而非硬凑结果 |
| 端到端质量 | 用 Claude 作为调用方 agent 跑完整问答,LLM-as-Judge 评最终回答(含 SKILL.md 是否被正确遵循) |
| 多语一致性 | 同一问题 sv/en/zh 检索,比对命中规则集合的一致性 |
| 响应速度 | tool 调用 P50/P90 |

CI 门禁:检索正确性 + 引用准确率 + 版本链正确性三项核心子集,回归则阻断合并。

## 8. 风险与对策

- **答错的后果真实存在** → 结构化返回不自由生成 + 强制出处 + 无数据即明说 + tool 描述内置边界声明,四道防线全是硬约束
- **调用方 agent 乱发挥** → SKILL.md 约定呈现格式;端到端评测覆盖"agent 拿到正确数据后是否答对"
- **官方网站改版导致抓取失效** → 解析器按源站模块化,监控抓取成功率,失败即告警
- **规则理解偏差(瑞典语法律文本)** → 双语对照校验;高频问题人工精标答案作为回归锚点
- **维护成本** → 增量 + 版本化架构把日常维护压缩为"看告警";数据源按 P0→P2 严格排序,不贪多

## 9. 技术栈

| 层 | 选型 |
|----|------|
| MCP | 官方 Python SDK / FastMCP,stdio + streamable HTTP 双形态 |
| 后端内核 | FastAPI + Pydantic v2 + SQLAlchemy 2.0 async + Alembic |
| Python 工具链 | uv + ruff + pytest |
| 抓取与解析 | httpx + 条件请求缓存;markitdown(快速通道)+ MinerU / 解析 API(重型通道) |
| 存储 | Postgres + pgvector(规则版本库 + 向量)、Redis(队列缓存) |
| 检索 | bge-m3 多语言向量 + BM25 + reranker |
| 任务 | ARQ 定时增量抓取 |
| 可观测 | Langfuse(端到端评测链路)+ OpenTelemetry(服务指标) |
| 部署 | docker-compose + Caddy + Hetzner VPS |
| 评测 | pytest + 自建 runner + LLM-as-Judge,GitHub Actions 门禁 |

与 Kollega 相比砍掉了整个前端栈(Next.js/shadcn/Vercel AI SDK),新增 MCP SDK。两个项目共享后端基建,Vistas 的差异化集中在:**多语言 embedding、时间维度数据模型、MCP 分发**。
